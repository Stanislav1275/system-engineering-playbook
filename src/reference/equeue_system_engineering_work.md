# Сервис электронных очередей и предварительной записи заявителей на приём

## 0. О работе

**Тема:** Сервис электронных очередей и предварительной записи заявителей на приём в многофункциональном центре.

**Назначение системы:** система предназначена для организации предварительной записи заявителей на услуги, управления расписанием приёма, отображения занятости слотов, поддержки электронной очереди в холле, уведомления заявителей и предоставления руководителю данных о загрузке подразделения.

**Тип системы:** информационная система для организаций с очным обслуживанием граждан.

**Основной контекст применения:** многофункциональный центр предоставления государственных и муниципальных услуг.

**System of Interest (SoI):** сервис электронных очередей и записи, включающий пользовательский интерфейс заявителя, рабочие места сотрудников, сервис записи, сервис расписаний, сервис очереди, модуль уведомлений, отчётность, журнал аудита и внутренние API.

---

## 1. Контекст и границы SoI

### 1.1. Цель системы

Целью системы является снижение нагрузки на регистраторов и уменьшение времени ожидания заявителей за счёт предварительной записи на услуги, автоматизированного управления временными слотами, поддержки электронной очереди и уведомления пользователей о создании, изменении или отмене записи.

Система должна обеспечивать корректное распределение заявителей по временным интервалам, предотвращать двойную запись на один и тот же слот, учитывать расписание подразделений и окон обслуживания, а также предоставлять сотрудникам актуальную информацию о текущей очереди и загрузке.

### 1.2. Внешние акторы

| Актор | Описание | Основной интерес |
|---|---|---|
| Заявитель | Физическое лицо, записывающееся на услугу | Быстро выбрать услугу, дату и время приёма, получить подтверждение и при необходимости отменить запись |
| Регистратор | Сотрудник, работающий с заявителями в день приёма | Видеть список записей, вызывать заявителей, менять статусы обслуживания |
| Администратор расписаний | Сотрудник, настраивающий графики работы подразделений и окон | Управлять услугами, окнами, длительностью слотов, перерывами и исключениями |
| Руководитель подразделения | Ответственный за качество и эффективность обслуживания | Анализировать загрузку окон, количество записей, отмен, неявок и среднее ожидание |
| Системный администратор | Ответственный за доступы, интеграции и эксплуатацию | Контролировать работоспособность системы, интеграции, журналирование и права доступа |
| Терминал электронной очереди | Устройство в холле МФЦ | Выдавать талоны заявителям, которые пришли без предварительной записи или подтверждают прибытие |
| Информационное табло | Устройство отображения очереди | Показывать номера талонов, окна обслуживания и статусы вызова |

### 1.3. Внешние системы

| Внешняя система | Назначение интеграции | Тип взаимодействия |
|---|---|---|
| SSO / IdP | Аутентификация сотрудников и проверка ролей | OAuth2/OIDC, JWT |
| Сервис уведомлений | Отправка SMS, e-mail или push-уведомлений | REST API или очередь сообщений |
| Внешний календарь / система расписаний | Синхронизация графиков работы и исключений | REST API, импорт/экспорт событий |
| Терминалы в холле | Получение талонов и подтверждение прибытия | REST API |
| Информационные табло | Отображение текущих вызовов и очереди | REST API/WebSocket |
| Система мониторинга | Сбор метрик доступности, ошибок и производительности | Metrics endpoint, logs |
| BI / аналитика | Получение агрегированных данных о загрузке | Экспорт отчётов/API |

### 1.4. Границы системы

#### Внутри SoI

В состав системы входят:

- публичный веб-интерфейс заявителя;
- интерфейс сотрудника и администратора расписаний;
- API Gateway / Backend API;
- сервис записи на приём;
- сервис расписаний и расчёта слотов;
- сервис электронной очереди;
- модуль уведомлений;
- модуль отчётности и аудита;
- основная база данных;
- кэш и механизм краткоживущих блокировок;
- очередь сообщений для событий записи и уведомлений;
- API для терминалов и табло.

#### Вне SoI

За пределами системы находятся:

- корпоративный SSO / IdP;
- внешний SMS/e-mail/push-шлюз;
- внешняя календарная система;
- физическое оборудование терминалов и табло;
- внешние средства мониторинга;
- BI-система организации;
- организационные регламенты МФЦ;
- фактическая работа сотрудников с заявителями вне интерфейса системы.

### 1.6. Допущения и ограничения

1. Система рассматривается в контексте одного МФЦ с возможностью расширения на несколько подразделений.
2. Заявитель может использовать веб-интерфейс без корпоративной учётной записи сотрудника.
3. Сотрудники входят в систему через SSO.
4. Уведомления отправляются через внешний сервис, который может быть временно недоступен.
5. Расписание может задаваться вручную администратором или синхронизироваться с внешним календарём.
6. Система должна сохранять корректность записи даже при одновременных запросах нескольких пользователей на один слот.
7. Терминалы и табло считаются внешним оборудованием; в SoI входят только API и логика взаимодействия с ними.
8. Персональные данные заявителей должны обрабатываться с учётом требований защиты информации и минимизации доступа.

## 2. CONOPS / OPSCON

### 2.1. Роли

| Роль | Описание деятельности | Основные операции |
|---|---|---|
| Заявитель | Пользуется системой для предварительной записи или отмены записи | Просмотр услуг, выбор слота, создание записи, отмена, перенос |
| Регистратор | Работает с текущим потоком заявителей | Просмотр списка, вызов, смена статуса, отметка неявки |
| Администратор расписаний | Настраивает доступность услуг и окон | Создание расписания, исключений, длительности услуг, перерывов |
| Руководитель подразделения | Контролирует эффективность обслуживания | Просмотр отчётов, загрузки, отмен, неявок, времени ожидания |
| Системный администратор | Поддерживает работоспособность системы | Контроль интеграций, доступов, журналов и метрик |
| Терминал | Автоматизированно выдаёт талон или подтверждает прибытие | Создание талона, поиск записи, передача события прибытия |

### 2.2. Сценарий UC-01. Предварительная запись заявителя на услугу

**Цель:** заявитель самостоятельно выбирает услугу, подразделение, дату и свободный слот приёма.

**Участники:** заявитель, Public Web UI, API Gateway, Schedule Service, Appointment Service, Notification Worker.

**Предусловия:**

- услуга заведена в системе;
- для подразделения настроено расписание;
- имеются свободные слоты;
- заявитель указал контактные данные.

**Основной поток:**

1. Заявитель открывает веб-интерфейс записи.
2. Система отображает список доступных услуг и подразделений.
3. Заявитель выбирает услугу, подразделение и дату.
4. Public Web UI отправляет запрос на получение доступных слотов.
5. Schedule Service рассчитывает свободные слоты на основании расписания, исключений и существующих записей.
6. Заявитель выбирает свободный слот.
7. Appointment Service создаёт запись в транзакции и фиксирует статус `confirmed`.
8. Система публикует событие `AppointmentCreated`.
9. Notification Worker отправляет подтверждение заявителю.
10. Заявитель видит номер записи, дату, время, услугу и подразделение.

**Исключения:**

- выбранный слот был занят другим пользователем до подтверждения;
- заявитель указал некорректные контактные данные;
- внешний сервис уведомлений недоступен;
- расписание на выбранную дату отсутствует.

**Режимы деградации:**

- если сервис уведомлений недоступен, запись создаётся, а уведомление ставится в очередь повторной отправки;
- если часть данных витрины занятости устарела, система повторно проверяет слот при создании записи;
- если внешний календарь недоступен, используется последняя успешно синхронизированная версия расписания с пометкой времени обновления.

**Результат:** запись создана, слот занят, заявитель получил подтверждение в интерфейсе и уведомление либо ожидает его повторной отправки.

### 2.3. Сценарий UC-02. Отмена записи заявителем

**Цель:** заявитель отменяет ранее созданную запись без обращения к регистратору.

**Участники:** заявитель, Public Web UI, Appointment Service, Notification Worker.

**Предусловия:**

- запись существует;
- запись находится в статусе `confirmed` или `waiting`;
- время приёма ещё не наступило либо политика системы допускает отмену.

**Основной поток:**

1. Заявитель открывает страницу записи по ссылке или в личном кабинете.
2. Система отображает параметры записи.
3. Заявитель выбирает действие «Отменить запись».
4. Система запрашивает подтверждение действия.
5. Appointment Service переводит запись в статус `cancelled`.
6. Слот становится доступным для других заявителей.
7. Система публикует событие `AppointmentCancelled`.
8. Notification Worker отправляет уведомление об отмене.

**Исключения:**

- запись не найдена;
- запись уже отменена;
- запись завершена или заявитель уже находится на приёме;
- пользователь пытается отменить чужую запись.

**Режимы деградации:**

- повторная отмена той же записи возвращает безопасный результат без повторного изменения данных;
- при недоступности уведомлений отмена всё равно фиксируется, а уведомление ставится в очередь.

**Результат:** запись отменена, слот освобождён, действие зафиксировано в истории.

### 2.4. Сценарий UC-03. Перенос записи на другой слот

**Цель:** заявитель выбирает другое время приёма без создания дублирующей записи.

**Участники:** заявитель, Public Web UI, Schedule Service, Appointment Service.

**Предусловия:**

- исходная запись существует;
- запись ещё не завершена;
- есть доступные альтернативные слоты.

**Основной поток:**

1. Заявитель открывает существующую запись.
2. Система показывает действие «Перенести запись».
3. Заявитель выбирает новую дату.
4. Schedule Service рассчитывает доступные слоты.
5. Заявитель выбирает новый слот.
6. Appointment Service в транзакции освобождает старый слот и закрепляет новый.
7. История записи сохраняет факт переноса.
8. Система отправляет уведомление с новым временем.

**Исключения:**

- новый слот уже занят;
- перенос запрещён правилами услуги;
- выбранная дата не входит в период доступной записи;
- исходная запись уже отменена.

**Режимы деградации:**

- если новый слот занял другой пользователь, система предлагает обновить список доступных слотов;
- если уведомление не отправлено, перенос всё равно считается выполненным после успешной транзакции.

**Результат:** у заявителя остаётся одна актуальная запись с новым временем.

### 2.5. Сценарий UC-04. Обслуживание заявителя регистратором

**Цель:** регистратор управляет приёмом заявителей в день оказания услуги.

**Участники:** регистратор, Staff UI, Queue Service, Appointment Service, табло.

**Предусловия:**

- регистратор авторизован через SSO;
- у регистратора есть права на выбранное подразделение;
- на текущий день есть записи или живая очередь.

**Основной поток:**

1. Регистратор открывает рабочее место.
2. Система отображает список заявителей на текущий день и талоны очереди.
3. Регистратор выбирает следующего заявителя.
4. Queue Service меняет статус талона на `called`.
5. Информационное табло отображает номер талона и окно обслуживания.
6. После явки заявителя регистратор переводит запись в статус `in_service`.
7. После завершения обслуживания регистратор переводит запись в статус `completed`.

**Исключения:**

- заявитель не явился;
- регистратор потерял сессию;
- табло недоступно;
- запись найдена, но услуга не соответствует окну обслуживания.

**Режимы деградации:**

- при недоступности табло регистратор может вызвать заявителя вручную;
- при временной недоступности SSO активная сессия сотрудника продолжает работу до истечения локального токена;
- при сбое обновления табло статус сохраняется в системе, а событие отправляется повторно.

**Результат:** обслуживание отражено в системе, статус записи обновлён, история действий сохранена.

### 2.6. Сценарий UC-05. Настройка расписания администратором

**Цель:** администратор задаёт услуги, окна обслуживания, рабочие интервалы, длительность слотов и исключения.

**Участники:** администратор расписаний, Admin UI, Schedule Service.

**Предусловия:**

- администратор авторизован;
- у администратора есть права на управление расписанием подразделения.

**Основной поток:**

1. Администратор открывает раздел управления расписанием.
2. Система отображает подразделения, услуги и окна обслуживания.
3. Администратор задаёт рабочие дни, интервалы, перерывы и длительность слота.
4. Администратор добавляет исключения: выходной, сокращённый день, закрытие окна.
5. Schedule Service проверяет конфликты расписания.
6. Система сохраняет расписание и пересчитывает доступность будущих слотов.
7. Изменение фиксируется в журнале аудита.

**Исключения:**

- расписание пересекается с уже существующими записями;
- указана некорректная длительность слота;
- администратор пытается удалить окно, на которое уже есть будущие записи.

**Режимы деградации:**

- при конфликте с существующими записями система запрещает изменение или требует явного решения: перенос записей, закрытие только свободных слотов, уведомление заявителей;
- при недоступности внешнего календаря изменения сохраняются локально и синхронизируются позже.

**Результат:** расписание сохранено, будущая витрина слотов обновлена.

### 2.7. Сценарий UC-06. Получение талона через терминал в холле

**Цель:** заявитель получает талон электронной очереди на месте или подтверждает прибытие по предварительной записи.

**Участники:** заявитель, терминал, Terminal API, Queue Service, табло.

**Предусловия:**

- терминал подключён к сети;
- подразделение открыто;
- Queue Service доступен.

**Основной поток:**

1. Заявитель выбирает на терминале услугу или вводит номер предварительной записи.
2. Терминал отправляет запрос в Terminal API.
3. Queue Service создаёт талон или связывает талон с существующей записью.
4. Терминал печатает номер талона.
5. Табло получает обновлённое состояние очереди.
6. Регистратор видит заявителя в списке ожидания.

**Исключения:**

- запись не найдена;
- заявитель пришёл слишком рано или слишком поздно;
- услуга недоступна для живой очереди;
- терминал не может распечатать талон.

**Режимы деградации:**

- при недоступности печати терминал отображает номер талона на экране;
- при недоступности терминала регистратор создаёт талон вручную через Staff UI;
- при недоступности табло вызов заявителя выполняется голосом или вручную.

**Результат:** заявитель получает место в очереди, а сотрудники видят его в рабочем списке.

### 2.8. Сценарий UC-07. Работа при недоступности сервиса уведомлений

**Цель:** система сохраняет возможность создания и изменения записей даже при сбое внешнего уведомительного шлюза.

**Участники:** Appointment Service, Message Broker, Notification Worker, Notification Gateway.

**Предусловия:**

- заявитель создаёт, отменяет или переносит запись;
- внешний сервис уведомлений временно недоступен.

**Основной поток:**

1. Appointment Service успешно выполняет основную операцию с записью.
2. Система публикует событие в Message Broker.
3. Notification Worker пытается отправить уведомление.
4. Notification Gateway возвращает ошибку или тайм-аут.
5. Notification Worker переводит задачу в статус `retry_pending`.
6. Система повторяет отправку по политике retry/backoff.
7. После успешной отправки задача получает статус `sent`.

**Исключения:**

- превышен лимит повторов;
- контакт заявителя некорректен;
- сообщение не может быть доставлено ни по одному каналу.

**Режимы деградации:**

- после превышения лимита повторов задача переводится в dead-letter queue;
- оператор или администратор видит список недоставленных уведомлений;
- заявитель всё равно видит актуальный статус записи в интерфейсе.

**Результат:** основная операция с записью не блокируется внешним сервисом уведомлений.

## 4. Архитектура системы

### 4.1. Общий архитектурный подход

Система строится как модульная клиент-серверная информационная система с выделенными функциональными компонентами: управление записью, управление расписанием, электронная очередь, уведомления, отчётность и аудит. Критичные операции, связанные с созданием записи на слот, выполняются транзакционно, так как основным архитектурным риском является конкурентная запись нескольких пользователей на одно и то же время.

Для уменьшения связности внешние уведомления выполняются асинхронно через очередь сообщений. Недоступность SMS/e-mail/push-шлюза не должна блокировать создание записи. Для сотрудников используется интеграция с SSO, а для заявителей допускается идентификация через контактные данные, ссылку на запись или личный кабинет.

### 4.2. C4 Context

Контекстная диаграмма должна показывать систему как единый объект и внешние взаимодействия:

- заявитель взаимодействует с публичным интерфейсом записи;
- сотрудники взаимодействуют с интерфейсом сотрудника и администратора;
- терминалы и табло взаимодействуют с API очереди;
- SSO используется для аутентификации сотрудников;
- сервис уведомлений отправляет сообщения заявителям;
- внешний календарь предоставляет или принимает данные расписания;
- система мониторинга получает метрики и логи.

### 4.5. Механизм защиты от двойной записи

Консистентность слотов обеспечивается несколькими уровнями защиты:

1. `Idempotency-Key` на операции создания записи, чтобы повторный запрос клиента не создавал дубликат.
2. Краткоживущая блокировка слота в Redis на время выполнения операции.
3. Транзакционное создание записи в PostgreSQL.
4. Уникальное ограничение на активную запись в пределах окна обслуживания и времени начала.
5. Повторная проверка доступности слота непосредственно перед фиксацией записи.

Пример уникального ограничения:

```sql
CREATE UNIQUE INDEX ux_active_appointment_slot
ON appointments (department_id, service_id, window_id, start_at)
WHERE status IN ('pending_confirmation', 'confirmed', 'waiting', 'called', 'in_service');
```

### 4.6. ADR

#### ADR-1. Транзакционное создание записи с уникальным ограничением на активный слот

**Контекст:** несколько заявителей могут одновременно выбрать один и тот же свободный слот.

**Решение:** создание записи выполняется в транзакции, а база данных содержит уникальное ограничение на активный слот.

**Альтернативы:** только Redis-lock; только проверка на уровне приложения; предварительная генерация всех слотов.

**Trade-off:** уникальное ограничение усложняет модель статусов, но даёт надёжную защиту от двойной записи даже при сбое прикладной логики.

#### ADR-2. Асинхронная отправка уведомлений через очередь сообщений

**Контекст:** уведомительный шлюз может быть временно недоступен.

**Решение:** после изменения записи система публикует событие, а Notification Worker отправляет уведомления асинхронно.

**Альтернативы:** синхронная отправка в рамках запроса; периодический batch без событий.

**Trade-off:** пользователь может получить уведомление с задержкой, но основная операция записи не блокируется внешним сервисом.

#### ADR-3. Разделение Schedule Service и Appointment Service

**Контекст:** расписание определяет потенциальную доступность, а запись фиксирует фактическую занятость.

**Решение:** выделить отдельные компоненты для расчёта слотов и управления записями.

**Альтернативы:** единый сервис расписаний и записей.

**Trade-off:** появляется межсервисное взаимодействие, но упрощается ответственность компонентов и тестирование.

#### ADR-4. Использование SSO для сотрудников

**Контекст:** сотрудники организации уже имеют корпоративные учётные записи и роли.

**Решение:** использовать корпоративный SSO для входа сотрудников и проверки ролей.

**Альтернативы:** локальные логины и пароли внутри системы.

**Trade-off:** система зависит от доступности SSO, но снижает риск разрозненного управления учётными записями.

#### ADR-5. Ведение журнала аудита действий сотрудников

**Контекст:** сотрудники работают с персональными данными и изменяют статусы обслуживания.

**Решение:** фиксировать действия сотрудников в AuditLog.

**Альтернативы:** хранить только технические логи приложения.

**Trade-off:** увеличивается объём хранимых данных, но повышается контролируемость и расследуемость спорных действий.

## 6. Риски

### 6.1. Шкала оценки

**Вероятность (P):**

- 1 — низкая;
- 2 — средняя;
- 3 — высокая.

**Влияние (I):**

- 1 — низкое;
- 2 — среднее;
- 3 — высокое.

**Приоритет:** `P × I`.

### 7.3. Тест-кейсы

#### TC-01. Создание записи на свободный слот

**Цель:** проверить, что заявитель может создать запись на доступное время.

**Предусловия:** услуга и расписание созданы, свободный слот существует.

**Данные:** услуга `Получение документов`, подразделение `МФЦ-1`, дата будущего рабочего дня.

**Шаги:**

1. Открыть публичный интерфейс записи.
2. Выбрать услугу и подразделение.
3. Запросить доступные слоты.
4. Выбрать свободный слот.
5. Ввести контактные данные заявителя.
6. Подтвердить запись.

**Ожидаемый результат:** запись создана со статусом `confirmed`, слот больше не отображается как свободный, заявитель видит подтверждение.

**Постусловия:** в базе данных существует запись с корректными атрибутами.

**Покрываемые требования:** SR-SH-1, SR-SH-2, SR-SYS-1, SR-SYS-2, SR-SYS-8.

#### TC-02. Попытка двойной записи на один слот

**Цель:** проверить защиту от конкурентного бронирования одного слота.

**Предусловия:** существует один свободный слот.

**Данные:** два параллельных запроса на `POST /appointments` с одинаковым `serviceId`, `departmentId`, `windowId`, `startAt`.

**Шаги:**

1. Одновременно отправить два запроса создания записи на один слот.
2. Проверить ответы API.
3. Проверить состояние базы данных.

**Ожидаемый результат:** одна запись создаётся успешно, второй запрос получает ошибку `409 SLOT_ALREADY_BOOKED` или аналогичную ошибку конфликта.

**Постусловия:** в базе данных существует не более одной активной записи на выбранный слот.

**Покрываемые требования:** SR-SYS-5, SR-SYS-6, NFR-1.

#### TC-03. Отмена записи и освобождение слота

**Цель:** проверить корректную отмену записи заявителем.

**Предусловия:** существует запись в статусе `confirmed`.

**Данные:** `appointmentId` существующей записи.

**Шаги:**

1. Открыть запись.
2. Выполнить запрос отмены.
3. Проверить статус записи.
4. Запросить список слотов на ту же дату.

**Ожидаемый результат:** запись получает статус `cancelled`, слот снова доступен для записи, публикуется событие отмены.

**Постусловия:** история записи содержит факт отмены.

**Покрываемые требования:** SR-SH-3, SR-SYS-11, SR-SYS-12, SR-SYS-13, SR-SYS-14.

#### TC-04. Перенос записи

**Цель:** проверить перенос записи на другой свободный слот.

**Предусловия:** существует активная запись и доступен другой свободный слот.

**Данные:** исходный `appointmentId`, новый `startAt`.

**Шаги:**

1. Открыть существующую запись.
2. _Запросить доступные альтернативные слоты._
3. Выбрать новый слот.
4. Подтвердить перенос.
5. Проверить данные записи.

**Ожидаемый результат:** запись получает новое время, старый слот освобождается, история изменения сохраняется.

**Постусловия:** у заявителя остаётся одна активная запись.

**Покрываемые требования:** SR-SH-4, SR-SYS-15, SR-SYS-16, SR-SYS-17.

#### TC-05. Получение списка свободных слотов

**Цель:** проверить расчёт витрины занятости.

**Предусловия:** настроено расписание, часть слотов занята.

**Данные:** дата с рабочим расписанием и несколькими существующими записями.

**Шаги:**

1. Выполнить `GET /slots`.
2. Проверить, что в ответе есть только будущие свободные слоты.
3. Проверить отсутствие занятых и нерабочих интервалов.

**Ожидаемый результат:** система возвращает корректный список свободных слотов.

**Постусловия:** состояние данных не изменяется.

**Покрываемые требования:** SR-SYS-3, SR-SYS-4, SR-SYS-13.

#### TC-06. Запрет записи на нерабочее время

**Цель:** проверить, что система не позволяет записаться вне расписания.

**Предусловия:** есть дата или интервал, закрытый исключением расписания.

**Данные:** слот в нерабочем интервале.

**Шаги:**

1. Попытаться создать запись на закрытый интервал.
2. Проверить ответ API.
3. Проверить отсутствие записи в базе данных.

**Ожидаемый результат:** система возвращает ошибку `422 SLOT_NOT_AVAILABLE`, запись не создаётся.

**Постусловия:** слот остаётся недоступным.

**Покрываемые требования:** SR-SYS-4, SR-SYS-17.

#### TC-07. Создание и изменение расписания администратором

**Цель:** проверить управление расписанием и исключениями.

**Предусловия:** администратор авторизован, подразделение и окно обслуживания существуют.

**Данные:** рабочие дни, интервалы, перерыв, исключение на конкретную дату.

**Шаги:**

1. Создать расписание окна.
2. Добавить перерыв.
3. Добавить исключение на конкретную дату.
4. Запросить слоты по дате исключения.
5. Попытаться удалить окно с будущими активными записями.

**Ожидаемый результат:** расписание сохраняется, исключение учитывается, удаление окна с активными записями запрещено.

**Постусловия:** изменения расписания зафиксированы в журнале аудита.

**Покрываемые требования:** SR-SH-7, SR-SYS-24, SR-SYS-25, SR-SYS-26, SR-SYS-27.

#### TC-08. Изменение статуса заявителя регистратором

**Цель:** проверить работу регистратора с текущей очередью.

**Предусловия:** регистратор авторизован, существует запись на текущий день.

**Данные:** `appointmentId`, статусы `waiting`, `called`, `in_service`, `completed`.

**Шаги:**

1. Открыть список записей на текущий день.
2. Перевести запись в статус `waiting`.
3. Вызвать заявителя и перевести в статус `called`.
4. Перевести в `in_service`.
5. Завершить обслуживание статусом `completed`.

**Ожидаемый результат:** все допустимые переходы выполняются, время каждого перехода фиксируется.

**Постусловия:** запись завершена, история статусов сохранена.

**Покрываемые требования:** SR-SH-5, SR-SH-6, SR-SYS-18, SR-SYS-19, SR-SYS-20.

#### TC-09. Получение талона через терминал

**Цель:** проверить создание талона электронной очереди.

**Предусловия:** терминал активен, услуга доступна для живой очереди.

**Данные:** `departmentId`, `serviceId`, `terminalRequestId`.

**Шаги:**

1. Отправить запрос `POST /queue/tickets`.
2. Проверить номер талона в ответе.
3. Проверить появление талона в списке регистратора.
4. Проверить отображение на табло после вызова.

**Ожидаемый результат:** талон создан, отображается в очереди и может быть вызван.

**Постусловия:** очередь содержит новый талон.

**Покрываемые требования:** SR-SH-11, SR-SYS-21, SR-SYS-22, SR-SYS-23.

#### TC-10. Работа при недоступности уведомлений

**Цель:** проверить деградацию при сбое уведомительного сервиса.

**Предусловия:** внешний уведомительный шлюз имитирует ошибку или тайм-аут.

**Данные:** новая запись на свободный слот.

**Шаги:**

1. Создать запись.
2. Проверить, что запись успешно создана.
3. Проверить состояние задачи уведомления.
4. Дождаться повторных попыток отправки.
5. Проверить перевод в DLQ после превышения лимита.

**Ожидаемый результат:** запись не откатывается из-за сбоя уведомлений, задача уведомления переходит в retry/DLQ по правилам.

**Постусловия:** в системе есть запись и задача уведомления с корректным статусом.

**Покрываемые требования:** SR-SYS-9, SR-SYS-10, SR-SYS-14, SR-SYS-34, SR-SYS-35, NFR-11.

#### TC-11. Проверка доступа через SSO и ролей

**Цель:** проверить разграничение доступа сотрудников.

**Предусловия:** настроена интеграция с SSO, существуют пользователи с ролями регистратора, администратора и руководителя.

**Данные:** JWT-токены разных ролей.

**Шаги:**

1. Выполнить вход сотрудника через SSO.
2. Попробовать открыть операции, доступные роли.
3. Попробовать выполнить операцию, недоступную роли.
4. Проверить ответ API и запись в журнале аудита.

**Ожидаемый результат:** доступные операции разрешены, недоступные возвращают `403 FORBIDDEN`.

**Постусловия:** попытки доступа зафиксированы.

**Покрываемые требования:** SR-SH-9, SR-SH-12, SR-SYS-32, SR-SYS-33, NFR-8.

#### TC-12. Нагрузочная проверка получения слотов и создания записи

**Цель:** проверить выполнение требований производительности.

**Предусловия:** тестовый стенд содержит расписания, услуги и записи, похожие на рабочую нагрузку.

**Данные:** 100 RPS для `GET /slots`, 50 RPS для `POST /appointments`.

**Шаги:**

1. Запустить нагрузочный тест на получение слотов.
2. Измерить p95 времени ответа.
3. Запустить нагрузочный тест на создание записи.
4. Измерить p95 времени ответа и долю ошибок.

**Ожидаемый результат:** `GET /slots` выполняется не более 500 мс в p95, `POST /appointments` не более 1 секунды в p95.

**Постусловия:** сформирован отчёт нагрузочного тестирования.

**Покрываемые требования:** SR-SYS-3, NFR-2, NFR-3.

#### TC-13. Проверка отчёта по загрузке

**Цель:** проверить отчётность для руководителя.

**Предусловия:** в системе есть данные за выбранный период.

**Данные:** период отчёта, подразделение, окна обслуживания.

**Шаги:**

1. Открыть раздел отчётов.
2. Выбрать период и подразделение.
3. Сформировать отчёт по загрузке.
4. Сравнить агрегаты с исходными данными в базе.

**Ожидаемый результат:** отчёт содержит количество записей, отмен, завершений, неявок и загрузку окон.

**Постусловия:** отчёт доступен руководителю.

**Покрываемые требования:** SR-SH-8, SR-SYS-28, SR-SYS-29, SR-SYS-30.

#### TC-14. Проверка журналирования действий сотрудника

**Цель:** проверить аудит операций с персональными данными и статусами.

**Предусловия:** сотрудник авторизован.

**Данные:** просмотр записи, изменение статуса, изменение расписания.

**Шаги:**

1. Выполнить просмотр записи сотрудником.
2. Изменить статус записи.
3. Изменить расписание.
4. Проверить записи в AuditLog.

**Ожидаемый результат:** журнал содержит сотрудника, роль, время, действие, тип объекта и идентификатор объекта.

**Постусловия:** аудит доступен для анализа.

**Покрываемые требования:** SR-SH-10, SR-SYS-16, SR-SYS-20, SR-SYS-31, NFR-7.

#### TC-15. Повторный запрос создания записи с Idempotency-Key

**Цель:** проверить защиту от повторного создания записи при повторной отправке запроса клиентом.

**Предусловия:** существует свободный слот.

**Данные:** два одинаковых запроса `POST /appointments` с одним и тем же `Idempotency-Key`.

**Шаги:**

1. Отправить первый запрос создания записи.
2. Не дожидаясь или имитируя потерю ответа, отправить повторный запрос с тем же `Idempotency-Key`.
3. Проверить ответы и состояние базы данных.

**Ожидаемый результат:** система возвращает один и тот же результат или безопасный повтор, не создавая вторую запись.

**Постусловия:** существует только одна запись, связанная с данным idempotency-key.

**Покрываемые требования:** SR-SYS-7.
