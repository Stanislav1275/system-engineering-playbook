## 7.3. Тест-кейсы

### TC-01. Создание записи на свободный слот

**Цель:** проверить, что заявитель может создать запись на доступное время.

**Предусловия:** услуга и расписание созданы, свободный слот существует.

**Данные:** услуга `Получение документов`, подразделение `МФЦ-1`, дата будущего рабочего дня.

**Шаги:**

1. Открыть публичный интерфейс записи.
2. Выбрать услугу и подразделение.
3. Запросить доступные слоты.
4. Выбрать свободный слот.
5. Ввести контактные данные заявителя.
6. Подтвердить запись.

**Ожидаемый результат:** запись создана со статусом `confirmed`, слот больше не отображается как свободный, заявитель видит подтверждение.

**Постусловия:** в базе данных существует запись с корректными атрибутами.

**Покрываемые требования:** SR-SH-1, SR-SH-2, SR-SYS-1, SR-SYS-2, SR-SYS-8.

### TC-02. Попытка двойной записи на один слот

**Цель:** проверить защиту от конкурентного бронирования одного слота.

**Предусловия:** существует один свободный слот.

**Данные:** два параллельных запроса на `POST /appointments` с одинаковым `serviceId`, `departmentId`, `windowId`, `startAt`.

**Шаги:**

1. Одновременно отправить два запроса создания записи на один слот.
2. Проверить ответы API.
3. Проверить состояние базы данных.

**Ожидаемый результат:** одна запись создаётся успешно, второй запрос получает ошибку `409 SLOT_ALREADY_BOOKED` или аналогичную ошибку конфликта.

**Постусловия:** в базе данных существует не более одной активной записи на выбранный слот.

**Покрываемые требования:** SR-SYS-5, SR-SYS-6, NFR-1.

### TC-03. Отмена записи и освобождение слота

**Цель:** проверить корректную отмену записи заявителем.

**Предусловия:** существует запись в статусе `confirmed`.

**Данные:** `appointmentId` существующей записи.

**Шаги:**

1. Открыть запись.
2. Выполнить запрос отмены.
3. Проверить статус записи.
4. Запросить список слотов на ту же дату.

**Ожидаемый результат:** запись получает статус `cancelled`, слот снова доступен для записи, публикуется событие отмены.

**Постусловия:** история записи содержит факт отмены.

**Покрываемые требования:** SR-SH-3, SR-SYS-11, SR-SYS-12, SR-SYS-13, SR-SYS-14.

### TC-04. Перенос записи

**Цель:** проверить перенос записи на другой свободный слот.

**Предусловия:** существует активная запись и доступен другой свободный слот.

**Данные:** исходный `appointmentId`, новый `startAt`.

**Шаги:**

1. Открыть существующую запись.
2. Запросить доступные альтернативные слоты.
3. Выбрать новый слот.
4. Подтвердить перенос.
5. Проверить данные записи.

**Ожидаемый результат:** запись получает новое время, старый слот освобождается, история изменения сохраняется.

**Постусловия:** у заявителя остаётся одна активная запись.

**Покрываемые требования:** SR-SH-4, SR-SYS-15, SR-SYS-16, SR-SYS-17.

### TC-05. Получение списка свободных слотов

**Цель:** проверить расчёт витрины занятости.

**Предусловия:** настроено расписание, часть слотов занята.

**Данные:** дата с рабочим расписанием и несколькими существующими записями.

**Шаги:**

1. Выполнить `GET /slots`.
2. Проверить, что в ответе есть только будущие свободные слоты.
3. Проверить отсутствие занятых и нерабочих интервалов.

**Ожидаемый результат:** система возвращает корректный список свободных слотов.

**Постусловия:** состояние данных не изменяется.

**Покрываемые требования:** SR-SYS-3, SR-SYS-4, SR-SYS-13.

### TC-06. Запрет записи на нерабочее время

**Цель:** проверить, что система не позволяет записаться вне расписания.

**Предусловия:** есть дата или интервал, закрытый исключением расписания.

**Данные:** слот в нерабочем интервале.

**Шаги:**

1. Попытаться создать запись на закрытый интервал.
2. Проверить ответ API.
3. Проверить отсутствие записи в базе данных.

**Ожидаемый результат:** система возвращает ошибку `422 SLOT_NOT_AVAILABLE`, запись не создаётся.

**Постусловия:** слот остаётся недоступным.

**Покрываемые требования:** SR-SYS-4, SR-SYS-17.

### TC-07. Создание и изменение расписания администратором

**Цель:** проверить управление расписанием и исключениями.

**Предусловия:** администратор авторизован, подразделение и окно обслуживания существуют.

**Данные:** рабочие дни, интервалы, перерыв, исключение на конкретную дату.

**Шаги:**

1. Создать расписание окна.
2. Добавить перерыв.
3. Добавить исключение на конкретную дату.
4. Запросить слоты по дате исключения.
5. Попытаться удалить окно с будущими активными записями.

**Ожидаемый результат:** расписание сохраняется, исключение учитывается, удаление окна с активными записями запрещено.

**Постусловия:** изменения расписания зафиксированы в журнале аудита.

**Покрываемые требования:** SR-SH-7, SR-SYS-24, SR-SYS-25, SR-SYS-26, SR-SYS-27.

### TC-08. Изменение статуса заявителя регистратором

**Цель:** проверить работу регистратора с текущей очередью.

**Предусловия:** регистратор авторизован, существует запись на текущий день.

**Данные:** `appointmentId`, статусы `waiting`, `called`, `in_service`, `completed`.

**Шаги:**

1. Открыть список записей на текущий день.
2. Перевести запись в статус `waiting`.
3. Вызвать заявителя и перевести в статус `called`.
4. Перевести в `in_service`.
5. Завершить обслуживание статусом `completed`.

**Ожидаемый результат:** все допустимые переходы выполняются, время каждого перехода фиксируется.

**Постусловия:** запись завершена, история статусов сохранена.

**Покрываемые требования:** SR-SH-5, SR-SH-6, SR-SYS-18, SR-SYS-19, SR-SYS-20.

### TC-09. Получение талона через терминал

**Цель:** проверить создание талона электронной очереди.

**Предусловия:** терминал активен, услуга доступна для живой очереди.

**Данные:** `departmentId`, `serviceId`, `terminalRequestId`.

**Шаги:**

1. Отправить запрос `POST /queue/tickets`.
2. Проверить номер талона в ответе.
3. Проверить появление талона в списке регистратора.
4. Проверить отображение на табло после вызова.

**Ожидаемый результат:** талон создан, отображается в очереди и может быть вызван.

**Постусловия:** очередь содержит новый талон.

**Покрываемые требования:** SR-SH-11, SR-SYS-21, SR-SYS-22, SR-SYS-23.

### TC-10. Работа при недоступности уведомлений

**Цель:** проверить деградацию при сбое уведомительного сервиса.

**Предусловия:** внешний уведомительный шлюз имитирует ошибку или тайм-аут.

**Данные:** новая запись на свободный слот.

**Шаги:**

1. Создать запись.
2. Проверить, что запись успешно создана.
3. Проверить состояние задачи уведомления.
4. Дождаться повторных попыток отправки.
5. Проверить перевод в DLQ после превышения лимита.

**Ожидаемый результат:** запись не откатывается из-за сбоя уведомлений, задача уведомления переходит в retry/DLQ по правилам.

**Постусловия:** в системе есть запись и задача уведомления с корректным статусом.

**Покрываемые требования:** SR-SYS-9, SR-SYS-10, SR-SYS-14, SR-SYS-34, SR-SYS-35, NFR-11.

### TC-11. Проверка доступа через SSO и ролей

**Цель:** проверить разграничение доступа сотрудников.

**Предусловия:** настроена интеграция с SSO, существуют пользователи с ролями регистратора, администратора и руководителя.

**Данные:** JWT-токены разных ролей.

**Шаги:**

1. Выполнить вход сотрудника через SSO.
2. Попробовать открыть операции, доступные роли.
3. Попробовать выполнить операцию, недоступную роли.
4. Проверить ответ API и запись в журнале аудита.

**Ожидаемый результат:** доступные операции разрешены, недоступные возвращают `403 FORBIDDEN`.

**Постусловия:** попытки доступа зафиксированы.

**Покрываемые требования:** SR-SH-9, SR-SH-12, SR-SYS-32, SR-SYS-33, NFR-8.

### TC-12. Нагрузочная проверка получения слотов и создания записи

**Цель:** проверить выполнение требований производительности.

**Предусловия:** тестовый стенд содержит расписания, услуги и записи, похожие на рабочую нагрузку.

**Данные:** 100 RPS для `GET /slots`, 50 RPS для `POST /appointments`.

**Шаги:**

1. Запустить нагрузочный тест на получение слотов.
2. Измерить p95 времени ответа.
3. Запустить нагрузочный тест на создание записи.
4. Измерить p95 времени ответа и долю ошибок.

**Ожидаемый результат:** `GET /slots` выполняется не более 500 мс в p95, `POST /appointments` не более 1 секунды в p95.

**Постусловия:** сформирован отчёт нагрузочного тестирования.

**Покрываемые требования:** SR-SYS-3, NFR-2, NFR-3.

### TC-13. Проверка отчёта по загрузке

**Цель:** проверить отчётность для руководителя.

**Предусловия:** в системе есть данные за выбранный период.

**Данные:** период отчёта, подразделение, окна обслуживания.

**Шаги:**

1. Открыть раздел отчётов.
2. Выбрать период и подразделение.
3. Сформировать отчёт по загрузке.
4. Сравнить агрегаты с исходными данными в базе.

**Ожидаемый результат:** отчёт содержит количество записей, отмен, завершений, неявок и загрузку окон.

**Постусловия:** отчёт доступен руководителю.

**Покрываемые требования:** SR-SH-8, SR-SYS-28, SR-SYS-29, SR-SYS-30.

### TC-14. Проверка журналирования действий сотрудника

**Цель:** проверить аудит операций с персональными данными и статусами.

**Предусловия:** сотрудник авторизован.

**Данные:** просмотр записи, изменение статуса, изменение расписания.

**Шаги:**

1. Выполнить просмотр записи сотрудником.
2. Изменить статус записи.
3. Изменить расписание.
4. Проверить записи в AuditLog.

**Ожидаемый результат:** журнал содержит сотрудника, роль, время, действие, тип объекта и идентификатор объекта.

**Постусловия:** аудит доступен для анализа.

**Покрываемые требования:** SR-SH-10, SR-SYS-16, SR-SYS-20, SR-SYS-31, NFR-7.

### TC-15. Повторный запрос создания записи с Idempotency-Key

**Цель:** проверить защиту от повторного создания записи при повторной отправке запроса клиентом.

**Предусловия:** существует свободный слот.

**Данные:** два одинаковых запроса `POST /appointments` с одним и тем же `Idempotency-Key`.

**Шаги:**

1. Отправить первый запрос создания записи.
2. Не дожидаясь или имитируя потерю ответа, отправить повторный запрос с тем же `Idempotency-Key`.
3. Проверить ответы и состояние базы данных.

**Ожидаемый результат:** система возвращает один и тот же результат или безопасный повтор, не создавая вторую запись.

**Постусловия:** существует только одна запись, связанная с данным idempotency-key.

**Покрываемые требования:** SR-SYS-7.
