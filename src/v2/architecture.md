# Архитектура системы

## 4.1. Общий архитектурный подход

Система строится как модульная клиент-серверная информационная система с выделенными функциональными компонентами: управление записью, управление расписанием, электронная очередь, уведомления, отчётность и аудит. Критичные операции, связанные с созданием записи на слот, выполняются транзакционно, так как основным архитектурным риском является конкурентная запись нескольких пользователей на одно и то же время.

Для уменьшения связности внешние уведомления выполняются асинхронно через очередь сообщений. Недоступность SMS/e-mail/push-шлюза не должна блокировать создание записи. Для сотрудников используется интеграция с SSO, а для заявителей допускается идентификация через контактные данные, ссылку на запись или личный кабинет.

## 4.2. C4 Context

Контекстная диаграмма показывает систему как единый объект и внешние взаимодействия:

- заявитель взаимодействует с публичным интерфейсом записи;
- сотрудники взаимодействуют с интерфейсом сотрудника и администратора;
- терминалы и табло взаимодействуют с API очереди;
- SSO используется для аутентификации сотрудников;
- сервис уведомлений отправляет сообщения заявителям;
- внешний календарь предоставляет или принимает данные расписания;
- система мониторинга получает метрики и логи.

## 4.5. Механизм защиты от двойной записи

Консистентность слотов обеспечивается несколькими уровнями защиты:

1. `Idempotency-Key` на операции создания записи, чтобы повторный запрос клиента не создавал дубликат.
2. Краткоживущая блокировка слота в Redis на время выполнения операции.
3. Транзакционное создание записи в PostgreSQL.
4. Уникальное ограничение на активную запись в пределах окна обслуживания и времени начала.
5. Повторная проверка доступности слота непосредственно перед фиксацией записи.

```sql
CREATE UNIQUE INDEX ux_active_appointment_slot
ON appointments (department_id, service_id, window_id, start_at)
WHERE status IN ('pending_confirmation', 'confirmed', 'waiting', 'called', 'in_service');
```

## 4.6. ADR

### ADR-1. Транзакционное создание записи с уникальным ограничением на активный слот

**Контекст:** несколько заявителей могут одновременно выбрать один и тот же свободный слот.

**Решение:** создание записи выполняется в транзакции, а база данных содержит уникальное ограничение на активный слот.

**Альтернативы:** только Redis-lock; только проверка на уровне приложения; предварительная генерация всех слотов.

**Trade-off:** уникальное ограничение усложняет модель статусов, но даёт надёжную защиту от двойной записи даже при сбое прикладной логики.

### ADR-2. Асинхронная отправка уведомлений через очередь сообщений

**Контекст:** уведомительный шлюз может быть временно недоступен.

**Решение:** после изменения записи система публикует событие, а Notification Worker отправляет уведомления асинхронно.

**Альтернативы:** синхронная отправка в рамках запроса; периодический batch без событий.

**Trade-off:** пользователь может получить уведомление с задержкой, но основная операция записи не блокируется внешним сервисом.

### ADR-3. Разделение Schedule Service и Appointment Service

**Контекст:** расписание определяет потенциальную доступность, а запись фиксирует фактическую занятость.

**Решение:** выделить отдельные компоненты для расчёта слотов и управления записями.

**Альтернативы:** единый сервис расписаний и записей.

**Trade-off:** появляется межсервисное взаимодействие, но упрощается ответственность компонентов и тестирование.

### ADR-4. Использование SSO для сотрудников

**Контекст:** сотрудники организации уже имеют корпоративные учётные записи и роли.

**Решение:** использовать корпоративный SSO для входа сотрудников и проверки ролей.

**Альтернативы:** локальные логины и пароли внутри системы.

**Trade-off:** система зависит от доступности SSO, но снижает риск разрозненного управления учётными записями.

### ADR-5. Ведение журнала аудита действий сотрудников

**Контекст:** сотрудники работают с персональными данными и изменяют статусы обслуживания.

**Решение:** фиксировать действия сотрудников в AuditLog.

**Альтернативы:** хранить только технические логи приложения.

**Trade-off:** увеличивается объём хранимых данных, но повышается контролируемость и расследуемость спорных действий.
