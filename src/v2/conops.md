# CONOPS / OPSCON

## 2.1. Роли

| Роль | Описание деятельности | Основные операции |
|---|---|---|
| Заявитель | Пользуется системой для предварительной записи или отмены записи | Просмотр услуг, выбор слота, создание записи, отмена, перенос |
| Регистратор | Работает с текущим потоком заявителей | Просмотр списка, вызов, смена статуса, отметка неявки |
| Администратор расписаний | Настраивает доступность услуг и окон | Создание расписания, исключений, длительности услуг, перерывов |
| Руководитель подразделения | Контролирует эффективность обслуживания | Просмотр отчётов, загрузки, отмен, неявок, времени ожидания |
| Системный администратор | Поддерживает работоспособность системы | Контроль интеграций, доступов, журналов и метрик |
| Терминал | Автоматизированно выдаёт талон или подтверждает прибытие | Создание талона, поиск записи, передача события прибытия |

## 2.2. Сценарий UC-01. Предварительная запись заявителя на услугу

**Цель:** заявитель самостоятельно выбирает услугу, подразделение, дату и свободный слот приёма.

**Участники:** заявитель, Public Web UI, API Gateway, Schedule Service, Appointment Service, Notification Worker.

**Предусловия:**

- услуга заведена в системе;
- для подразделения настроено расписание;
- имеются свободные слоты;
- заявитель указал контактные данные.

**Основной поток:**

1. Заявитель открывает веб-интерфейс записи.
2. Система отображает список доступных услуг и подразделений.
3. Заявитель выбирает услугу, подразделение и дату.
4. Public Web UI отправляет запрос на получение доступных слотов.
5. Schedule Service рассчитывает свободные слоты на основании расписания, исключений и существующих записей.
6. Заявитель выбирает свободный слот.
7. Appointment Service создаёт запись в транзакции и фиксирует статус `confirmed`.
8. Система публикует событие `AppointmentCreated`.
9. Notification Worker отправляет подтверждение заявителю.
10. Заявитель видит номер записи, дату, время, услугу и подразделение.

**Исключения:**

- выбранный слот был занят другим пользователем до подтверждения;
- заявитель указал некорректные контактные данные;
- внешний сервис уведомлений недоступен;
- расписание на выбранную дату отсутствует.

**Режимы деградации:**

- если сервис уведомлений недоступен, запись создаётся, а уведомление ставится в очередь повторной отправки;
- если часть данных витрины занятости устарела, система повторно проверяет слот при создании записи;
- если внешний календарь недоступен, используется последняя успешно синхронизированная версия расписания с пометкой времени обновления.

**Результат:** запись создана, слот занят, заявитель получил подтверждение в интерфейсе и уведомление либо ожидает его повторной отправки.

## 2.3. Сценарий UC-02. Отмена записи заявителем

**Цель:** заявитель отменяет ранее созданную запись без обращения к регистратору.

**Участники:** заявитель, Public Web UI, Appointment Service, Notification Worker.

**Предусловия:**

- запись существует;
- запись находится в статусе `confirmed` или `waiting`;
- время приёма ещё не наступило либо политика системы допускает отмену.

**Основной поток:**

1. Заявитель открывает страницу записи по ссылке или в личном кабинете.
2. Система отображает параметры записи.
3. Заявитель выбирает действие «Отменить запись».
4. Система запрашивает подтверждение действия.
5. Appointment Service переводит запись в статус `cancelled`.
6. Слот становится доступным для других заявителей.
7. Система публикует событие `AppointmentCancelled`.
8. Notification Worker отправляет уведомление об отмене.

**Исключения:**

- запись не найдена;
- запись уже отменена;
- запись завершена или заявитель уже находится на приёме;
- пользователь пытается отменить чужую запись.

**Режимы деградации:**

- повторная отмена той же записи возвращает безопасный результат без повторного изменения данных;
- при недоступности уведомлений отмена всё равно фиксируется, а уведомление ставится в очередь.

**Результат:** запись отменена, слот освобождён, действие зафиксировано в истории.

## 2.4. Сценарий UC-03. Перенос записи на другой слот

**Цель:** заявитель выбирает другое время приёма без создания дублирующей записи.

**Участники:** заявитель, Public Web UI, Schedule Service, Appointment Service.

**Предусловия:**

- исходная запись существует;
- запись ещё не завершена;
- есть доступные альтернативные слоты.

**Основной поток:**

1. Заявитель открывает существующую запись.
2. Система показывает действие «Перенести запись».
3. Заявитель выбирает новую дату.
4. Schedule Service рассчитывает доступные слоты.
5. Заявитель выбирает новый слот.
6. Appointment Service в транзакции освобождает старый слот и закрепляет новый.
7. История записи сохраняет факт переноса.
8. Система отправляет уведомление с новым временем.

**Исключения:**

- новый слот уже занят;
- перенос запрещён правилами услуги;
- выбранная дата не входит в период доступной записи;
- исходная запись уже отменена.

**Режимы деградации:**

- если новый слот занял другой пользователь, система предлагает обновить список доступных слотов;
- если уведомление не отправлено, перенос всё равно считается выполненным после успешной транзакции.

**Результат:** у заявителя остаётся одна актуальная запись с новым временем.

## 2.5. Сценарий UC-04. Обслуживание заявителя регистратором

**Цель:** регистратор управляет приёмом заявителей в день оказания услуги.

**Участники:** регистратор, Staff UI, Queue Service, Appointment Service, табло.

**Предусловия:**

- регистратор авторизован через SSO;
- у регистратора есть права на выбранное подразделение;
- на текущий день есть записи или живая очередь.

**Основной поток:**

1. Регистратор открывает рабочее место.
2. Система отображает список заявителей на текущий день и талоны очереди.
3. Регистратор выбирает следующего заявителя.
4. Queue Service меняет статус талона на `called`.
5. Информационное табло отображает номер талона и окно обслуживания.
6. После явки заявителя регистратор переводит запись в статус `in_service`.
7. После завершения обслуживания регистратор переводит запись в статус `completed`.

**Исключения:**

- заявитель не явился;
- регистратор потерял сессию;
- табло недоступно;
- запись найдена, но услуга не соответствует окну обслуживания.

**Режимы деградации:**

- при недоступности табло регистратор может вызвать заявителя вручную;
- при временной недоступности SSO активная сессия сотрудника продолжает работу до истечения локального токена;
- при сбое обновления табло статус сохраняется в системе, а событие отправляется повторно.

**Результат:** обслуживание отражено в системе, статус записи обновлён, история действий сохранена.

## 2.6. Сценарий UC-05. Настройка расписания администратором

**Цель:** администратор задаёт услуги, окна обслуживания, рабочие интервалы, длительность слотов и исключения.

**Участники:** администратор расписаний, Admin UI, Schedule Service.

**Предусловия:**

- администратор авторизован;
- у администратора есть права на управление расписанием подразделения.

**Основной поток:**

1. Администратор открывает раздел управления расписанием.
2. Система отображает подразделения, услуги и окна обслуживания.
3. Администратор задаёт рабочие дни, интервалы, перерывы и длительность слота.
4. Администратор добавляет исключения: выходной, сокращённый день, закрытие окна.
5. Schedule Service проверяет конфликты расписания.
6. Система сохраняет расписание и пересчитывает доступность будущих слотов.
7. Изменение фиксируется в журнале аудита.

**Исключения:**

- расписание пересекается с уже существующими записями;
- указана некорректная длительность слота;
- администратор пытается удалить окно, на которое уже есть будущие записи.

**Режимы деградации:**

- при конфликте с существующими записями система запрещает изменение или требует явного решения: перенос записей, закрытие только свободных слотов, уведомление заявителей;
- при недоступности внешнего календаря изменения сохраняются локально и синхронизируются позже.

**Результат:** расписание сохранено, будущая витрина слотов обновлена.

## 2.7. Сценарий UC-06. Получение талона через терминал в холле

**Цель:** заявитель получает талон электронной очереди на месте или подтверждает прибытие по предварительной записи.

**Участники:** заявитель, терминал, Terminal API, Queue Service, табло.

**Предусловия:**

- терминал подключён к сети;
- подразделение открыто;
- Queue Service доступен.

**Основной поток:**

1. Заявитель выбирает на терминале услугу или вводит номер предварительной записи.
2. Терминал отправляет запрос в Terminal API.
3. Queue Service создаёт талон или связывает талон с существующей записью.
4. Терминал печатает номер талона.
5. Табло получает обновлённое состояние очереди.
6. Регистратор видит заявителя в списке ожидания.

**Исключения:**

- запись не найдена;
- заявитель пришёл слишком рано или слишком поздно;
- услуга недоступна для живой очереди;
- терминал не может распечатать талон.

**Режимы деградации:**

- при недоступности печати терминал отображает номер талона на экране;
- при недоступности терминала регистратор создаёт талон вручную через Staff UI;
- при недоступности табло вызов заявителя выполняется голосом или вручную.

**Результат:** заявитель получает место в очереди, а сотрудники видят его в рабочем списке.

## 2.8. Сценарий UC-07. Работа при недоступности сервиса уведомлений

**Цель:** система сохраняет возможность создания и изменения записей даже при сбое внешнего уведомительного шлюза.

**Участники:** Appointment Service, Message Broker, Notification Worker, Notification Gateway.

**Предусловия:**

- заявитель создаёт, отменяет или переносит запись;
- внешний сервис уведомлений временно недоступен.

**Основной поток:**

1. Appointment Service успешно выполняет основную операцию с записью.
2. Система публикует событие в Message Broker.
3. Notification Worker пытается отправить уведомление.
4. Notification Gateway возвращает ошибку или тайм-аут.
5. Notification Worker переводит задачу в статус `retry_pending`.
6. Система повторяет отправку по политике retry/backoff.
7. После успешной отправки задача получает статус `sent`.

**Исключения:**

- превышен лимит повторов;
- контакт заявителя некорректен;
- сообщение не может быть доставлено ни по одному каналу.

**Режимы деградации:**

- после превышения лимита повторов задача переводится в dead-letter queue;
- оператор или администратор видит список недоставленных уведомлений;
- заявитель всё равно видит актуальный статус записи в интерфейсе.

**Результат:** основная операция с записью не блокируется внешним сервисом уведомлений.
