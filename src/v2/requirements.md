# Требования

## 3.1. Stakeholder requirements

| ID | Заинтересованная сторона | Требование | Метод проверки |
|---|---|---|---|
| SR-SH-1 | Заявитель | Система должна позволять заявителю самостоятельно выбрать услугу, подразделение, дату и свободное время приёма. | D, T |
| SR-SH-2 | Заявитель | Система должна предоставлять заявителю подтверждение записи с датой, временем, услугой и подразделением. | D, T |
| SR-SH-3 | Заявитель | Система должна позволять заявителю отменить запись без обращения к сотруднику. | D, T |
| SR-SH-4 | Заявитель | Система должна позволять заявителю перенести запись на другой доступный слот. | D, T |
| SR-SH-5 | Регистратор | Система должна предоставлять регистратору актуальный список заявителей на текущий день. | D, T |
| SR-SH-6 | Регистратор | Система должна позволять регистратору вызывать заявителя и изменять статус обслуживания. | D, T |
| SR-SH-7 | Администратор расписаний | Система должна позволять администратору управлять услугами, окнами обслуживания, длительностью слотов и исключениями. | D, I, T |
| SR-SH-8 | Руководитель подразделения | Система должна предоставлять руководителю данные о загрузке окон, количестве записей, отмен и неявок. | D, A |
| SR-SH-9 | Системный администратор | Система должна поддерживать вход сотрудников через корпоративный механизм аутентификации. | I, T |
| SR-SH-10 | Системный администратор | Система должна обеспечивать журналирование действий сотрудников с персональными данными. | I, A, T |
| SR-SH-11 | Организация | Система должна снижать вероятность образования неуправляемой живой очереди за счёт предварительной записи и электронной очереди. | A, D |
| SR-SH-12 | Организация | Система должна обеспечивать защиту персональных данных заявителей и разграничение доступа по ролям. | I, T, A |

## 3.2. System requirements

| ID | Требование | Метод проверки |
|---|---|---|
| SR-SYS-1 | Система должна предоставлять список услуг, доступных в выбранном подразделении. | T |
| SR-SYS-2 | Система должна предоставлять список подразделений, в которых доступна выбранная услуга. | T |
| SR-SYS-3 | Система должна рассчитывать доступные слоты на основании расписания, длительности услуги, перерывов, исключений и уже существующих записей. | T, A |
| SR-SYS-4 | Система должна отображать только те слоты, которые находятся в будущем и не заняты активными записями. | T |
| SR-SYS-5 | Система должна запрещать создание более одной активной записи на один и тот же слот одного окна обслуживания. | T, I |
| SR-SYS-6 | Система должна использовать транзакционную проверку доступности слота при создании записи. | I, T |
| SR-SYS-7 | Система должна обеспечивать идемпотентность операции создания записи: повторный запрос с теми же параметрами не должен создавать дублирующую запись. | I, T |
| SR-SYS-8 | Система должна сохранять запись с уникальным идентификатором, статусом, услугой, подразделением, окном обслуживания, временем начала и окончания. | T, I |
| SR-SYS-9 | Система должна отправлять событие `AppointmentCreated` после успешного создания записи. | T |
| SR-SYS-10 | Система должна отправлять уведомление заявителю после создания записи. | T |
| SR-SYS-11 | Система должна позволять отменить запись в статусах `confirmed` и `waiting`. | T |
| SR-SYS-12 | Система должна переводить отменённую запись в статус `cancelled` без физического удаления из базы данных. | T, I |
| SR-SYS-13 | Система должна освобождать слот после отмены записи. | T |
| SR-SYS-14 | Система должна публиковать событие `AppointmentCancelled` после отмены записи. | T |
| SR-SYS-15 | Система должна позволять перенести запись на другой доступный слот. | T |
| SR-SYS-16 | При переносе записи система должна сохранять историю изменения времени приёма. | T, A |
| SR-SYS-17 | Система должна запрещать перенос записи на занятый или нерабочий слот. | T |
| SR-SYS-18 | Система должна предоставлять регистратору список записей и талонов очереди на выбранную дату. | T |
| SR-SYS-19 | Система должна позволять регистратору менять статус записи: `waiting`, `called`, `in_service`, `completed`, `no_show`. | T |
| SR-SYS-20 | Система должна фиксировать время каждого изменения статуса записи. | T, A |
| SR-SYS-21 | Система должна отображать на табло номер талона, окно обслуживания и статус вызова. | D, T |
| SR-SYS-22 | Система должна позволять создать талон электронной очереди через Terminal API. | T |
| SR-SYS-23 | Система должна позволять связать талон с предварительной записью при подтверждении прибытия заявителя. | T |
| SR-SYS-24 | Система должна позволять администратору создать расписание для окна обслуживания. | T |
| SR-SYS-25 | Система должна позволять администратору создавать исключения расписания: выходной, сокращённый день, закрытие окна, дополнительный рабочий интервал. | T |
| SR-SYS-26 | Система должна проверять конфликт изменений расписания с будущими активными записями. | T |
| SR-SYS-27 | Система должна запрещать удаление окна обслуживания, если на него существуют будущие активные записи. | T |
| SR-SYS-28 | Система должна предоставлять руководителю отчёт по загрузке окон за выбранный период. | T, A |
| SR-SYS-29 | Система должна рассчитывать количество созданных, отменённых и завершённых записей за период. | T, A |
| SR-SYS-30 | Система должна рассчитывать количество неявок заявителей за период. | T, A |
| SR-SYS-31 | Система должна журналировать действия сотрудников: вход, просмотр записи, изменение статуса, изменение расписания. | T, A, I |
| SR-SYS-32 | Система должна разграничивать доступ по ролям: заявитель, регистратор, администратор расписаний, руководитель, системный администратор. | T, I |
| SR-SYS-33 | Система должна проверять токен сотрудника через SSO или локально валидировать ранее выданный токен. | T, I |
| SR-SYS-34 | Система должна ставить уведомления в очередь повторной отправки при временной недоступности внешнего уведомительного сервиса. | T |
| SR-SYS-35 | Система должна переводить уведомление в dead-letter queue после превышения лимита повторов. | T, A |

## 3.3. NFR / требования качества

| ID | Категория | Требование | Метод проверки |
|---|---|---|---|
| NFR-1 | Консистентность | При одновременной попытке нескольких заявителей записаться на один слот система должна создавать не более одной активной записи. | T, I |
| NFR-2 | Производительность | Время ответа операции получения доступных слотов не должно превышать 500 мс в p95 при нагрузке 100 RPS. | T, A |
| NFR-3 | Производительность | Время создания записи не должно превышать 1 секунду в p95 при штатной нагрузке 50 RPS. | T, A |
| NFR-4 | Доступность | Доступность сервиса предварительной записи должна составлять не менее 99,5% в месяц, исключая согласованные окна обслуживания. | T, A, I |
| NFR-5 | Восстановление | RPO для основной базы данных не должен превышать 5 минут. | I, A |
| NFR-6 | Восстановление | RTO для восстановления сервиса после критического сбоя не должен превышать 30 минут. | I, D, A |
| NFR-7 | Безопасность | Все действия сотрудников с персональными данными должны журналироваться с указанием сотрудника, времени, действия и объекта. | I, T, A |
| NFR-8 | Безопасность | Доступ сотрудников к операциям системы должен определяться ролью и подразделением. | I, T |
| NFR-9 | Защита данных | Персональные данные заявителей должны передаваться только по защищённому каналу TLS. | I |
| NFR-10 | UX | Создание записи заявителем должно выполняться не более чем за 5 основных пользовательских шагов. | D, A |
| NFR-11 | Уведомления | Задание на отправку уведомления должно создаваться не позднее чем через 10 секунд после успешного изменения записи. | T, A |
| NFR-12 | Наблюдаемость | Система должна предоставлять метрики количества ошибок, времени ответа API, длины очереди уведомлений и количества недоставленных сообщений. | I, A |
