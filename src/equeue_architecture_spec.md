# Архитектура сервиса электронных очередей и предварительной записи

## 1. Назначение системы

Сервис предназначен для предварительной записи заявителей на услуги, управления слотами приёма, поддержки электронной очереди в холле, уведомлений о записи и формирования витрины занятости для сотрудников и руководителя подразделения.

Система рассматривается в контексте МФЦ. Аналогичная модель может быть адаптирована для поликлиники или университета, но в рамках работы основной предметной областью является очное обслуживание заявителей в подразделении МФЦ.

## 2. Границы системы

### Входит в границы SoI

- публикация списка подразделений и услуг;
- расчёт доступных слотов по графикам, окнам обслуживания и исключениям;
- создание, перенос и отмена записей;
- защита от двойного бронирования слота;
- управление статусами приёма заявителя;
- выдача талонов электронной очереди;
- интеграция с терминалом и информационным табло;
- постановка уведомлений в очередь отправки;
- рабочее место регистратора;
- панель администратора расписаний;
- отчёты по загрузке окон, отменам и неявкам;
- аудит действий сотрудников.

### Не входит в границы SoI

- хранение корпоративных учётных записей сотрудников;
- физическая отправка SMS, e-mail и push-уведомлений;
- выпуск и управление средствами электронной подписи;
- внешняя система учёта обращений или дел;
- эксплуатация физических терминалов и табло как оборудования.

## 3. Архитектура C4

В файле `equeue_c4_architecture.drawio` подготовлены две страницы:

1. **C1 Контекст** — показывает пользователей, внешние системы и основные потоки взаимодействия.
2. **C2 Контейнеры** — показывает внутренние контейнеры системы, хранилища, брокер сообщений и интеграции.

## 4. Основные контейнеры

| Контейнер | Назначение |
|---|---|
| Public Web UI | Интерфейс заявителя для выбора услуги, даты, слота, создания, отмены и переноса записи |
| Staff/Admin UI | Рабочее место регистратора, администратора расписаний и руководителя подразделения |
| Terminal API | API для терминалов и табло в холле |
| API Gateway / Backend API | Единая точка входа, REST API, валидация, RBAC, rate limit |
| Auth Adapter | Интеграция с SSO, проверка токенов и ролей сотрудников |
| Appointment Service | Создание, перенос, отмена записей, защита от двойного бронирования |
| Schedule Service | Услуги, окна обслуживания, графики, исключения, расчёт доступных слотов |
| Queue Service | Талоны электронной очереди, вызов заявителя, статусы обслуживания |
| Notification Worker | Асинхронная отправка уведомлений, ретраи, dead-letter queue |
| Reporting & Audit Service | Отчёты, аудит действий, метрики загрузки |
| PostgreSQL | Основное транзакционное хранилище |
| Redis | Краткоживущие блокировки слотов, idempotency-key, кэш доступных слотов |
| Message Broker | События AppointmentCreated, AppointmentCancelled, ReminderDue |

## 6. Основные потоки данных

### Поток 1. Предварительная запись заявителя

1. Заявитель открывает Public Web UI.
2. Public Web UI запрашивает список подразделений и услуг через API Gateway.
3. API Gateway обращается к Schedule Service.
4. Schedule Service рассчитывает доступные слоты по расписаниям, исключениям и существующим записям в PostgreSQL.
5. Заявитель выбирает слот и отправляет заявку на запись.
6. Appointment Service проверяет доступность слота.
7. Appointment Service устанавливает краткоживущую блокировку в Redis.
8. Appointment Service в транзакции создаёт запись в PostgreSQL.
9. Уникальное ограничение БД не позволяет создать две активные записи на один и тот же слот.
10. Appointment Service публикует событие AppointmentCreated в Message Broker.
11. Notification Worker получает событие и отправляет уведомление через внешний шлюз.

### Поток 2. Отмена записи

1. Заявитель открывает запись.
2. Public Web UI отправляет запрос отмены.
3. Appointment Service проверяет статус записи.
4. Appointment Service переводит запись в статус cancelled.
5. Слот снова становится доступным при следующем расчёте доступности.
6. В Message Broker публикуется событие AppointmentCancelled.
7. Notification Worker отправляет уведомление об отмене.

### Поток 3. Обслуживание заявителя регистратором

1. Регистратор входит через Staff/Admin UI.
2. API Gateway проверяет токен через Auth Adapter и SSO.
3. Регистратор открывает список записей на текущий день.
4. Queue Service показывает ожидающих заявителей и талоны.
5. Регистратор вызывает заявителя к окну.
6. Queue Service меняет статус на called или in_service.
7. После завершения приёма статус меняется на completed.
8. Reporting & Audit Service фиксирует действие.

### Поток 4. Настройка расписания

1. Администратор расписаний входит через Staff/Admin UI.
2. API Gateway проверяет роль администратора.
3. Администратор создаёт или меняет график окна обслуживания.
4. Schedule Service валидирует график и исключения.
5. Schedule Service сохраняет изменения в PostgreSQL.
6. Reporting & Audit Service фиксирует изменение расписания.
7. При необходимости Schedule Service синхронизирует исключения с внешним календарём.

### Поток 5. Работа терминала в холле

1. Заявитель выбирает услугу на терминале.
2. Терминал отправляет запрос в Terminal API.
3. Terminal API передаёт команду в Queue Service.
4. Queue Service создаёт талон и сохраняет его в PostgreSQL.
5. Табло запрашивает состояние очереди.
6. Queue Service возвращает номера талонов и окна обслуживания.

### Поток 6. Деградация уведомлений

1. Appointment Service успешно создаёт запись.
2. Событие AppointmentCreated публикуется в брокер.
3. Notification Worker пытается отправить уведомление.
4. Если шлюз уведомлений недоступен, задача переводится в retry.
5. После превышения лимита повторов задача попадает в dead-letter queue.
6. Запись остаётся действительной, потому что создание записи не зависит от внешнего шлюза уведомлений.

## 7. Требования к консистентности слотов

Для защиты от двойного бронирования используются три уровня:

1. **Idempotency-Key** на запросе создания записи — защищает от повторной отправки формы.
2. **Краткоживущая блокировка слота в Redis** — снижает вероятность конкурентной гонки.
3. **Уникальное ограничение в PostgreSQL** — финальная гарантия, что активная запись на один слот будет только одна.

Рекомендуемое ограничение БД:

```sql
CREATE UNIQUE INDEX ux_active_appointment_slot
ON appointments (department_id, service_id, window_id, start_at)
WHERE status IN ('pending_confirmation', 'confirmed', 'waiting', 'called', 'in_service');
```

## 8. Swagger / OpenAPI

Основной API вынесен в файл `equeue_openapi.yaml`.

Ключевые группы эндпоинтов:

- `/services` — список услуг;
- `/departments` — список подразделений;
- `/slots` — витрина доступных слотов;
- `/appointments` — создание и просмотр записей;
- `/appointments/{appointmentId}/cancel` — отмена записи;
- `/appointments/{appointmentId}/reschedule` — перенос записи;
- `/registrar/appointments` — список записей для регистратора;
- `/registrar/appointments/{appointmentId}/status` — изменение статуса приёма;
- `/admin/schedules` — управление расписаниями;
- `/admin/schedule-exceptions` — исключения расписания;
- `/queue/tickets` — создание талона;
- `/queue/board/{departmentId}` — состояние табло;
- `/reports/load` — отчёт по загрузке.
